# Implement Phase

The **Implement** phase generates actual code using crew members—specialized AI agents that produce code, tests, and documentation.

## Purpose

Implement transforms technical designs into working code:

```
┌─────────────────────┐         ┌─────────────────────────┐
│  Architect Outputs  │         │   Implementation        │
│                     │         │                         │
│ - Design Document   │  ───▶   │ - Source Code           │
│ - Components        │         │ - Unit Tests            │
│ - API Contracts     │         │ - Integration Tests     │
│                     │         │ - Documentation         │
└─────────────────────┘         └─────────────────────────┘
```

## Running Implement

Unlike other SAIL phases, Implement uses **crew dispatch**:

```bash
# Dispatch the coder crew member
ldf crew dispatch coder

# Dispatch with options
ldf crew dispatch coder --reasoning high --max-tokens 16000

# Dispatch multiple crew members
ldf crew dispatch coder tester
```

### Coder Crew Member

The `coder` is the primary implementation crew member:

```bash
# Basic dispatch
ldf crew dispatch coder

# With higher reasoning for complex code
ldf crew dispatch coder --reasoning high

# With specific temperature
ldf crew dispatch coder --temperature 0.5
```

Coder produces:
- Implementation files based on component specs
- Inline documentation
- Type definitions

### Tester Crew Member

The `tester` generates tests based on TDD anchors:

```bash
# Generate tests
ldf crew dispatch tester

# Focus on specific anchor types
ldf crew dispatch tester --types unit,integration
```

Tester produces:
- Unit tests for each component
- Integration tests for API endpoints
- Test fixtures and mocks

## Dispatch Options

### Reasoning Level

Control how much the AI "thinks" before responding:

```bash
# Minimal reasoning (faster, cheaper)
ldf crew dispatch coder --reasoning none

# Balanced (default)
ldf crew dispatch coder --reasoning medium

# Maximum reasoning (slower, better for complex code)
ldf crew dispatch coder --reasoning high

# Extended reasoning for very complex tasks
ldf crew dispatch coder --reasoning xhigh
```

| Level | Use Case | Trade-off |
|-------|----------|-----------|
| `none` | Simple, straightforward code | Fast but may miss edge cases |
| `low` | Routine implementations | Good balance for common patterns |
| `medium` | Standard complexity (default) | Balanced quality/speed |
| `high` | Complex logic, algorithms | Higher quality, slower |
| `xhigh` | Critical, intricate code | Best quality, slowest |

### Token Limits

Control maximum output size:

```bash
# Smaller output
ldf crew dispatch coder --max-tokens 8000

# Larger output for complex features
ldf crew dispatch coder --max-tokens 32000
```

### Provider Selection

Use a specific provider:

```bash
# Use Claude
ldf crew dispatch coder --provider claude

# Use OpenAI
ldf crew dispatch coder --provider openai --model gpt-4o
```

## Guardrails

Generated code passes through guardrails before being written:

```bash
$ ldf crew dispatch coder
...
⚠️  Guardrail violation: security.no-hardcoded-secrets
   File: src/auth/config.ts, Line: 15
   Message: Potential hardcoded API key detected
   Suggestion: Use environment variables instead

Blocked: 1 error, 0 warnings
Fix issues and retry, or override with --force
```

See [Guardrails](/framework/guardrails/) for details.

## Re-running Implementation

You can re-run implementation to regenerate code:

```bash
# Regenerate all
ldf crew dispatch coder

# Regenerate specific components
ldf crew dispatch coder --components AuthService,TokenService

# Preserve existing files
ldf crew dispatch coder --no-overwrite
```

<Aside type="caution">
Re-running dispatch will overwrite existing files by default. Use version control to preserve your changes.
</Aside>

## Customizing Implementation

Configure default behavior in `.ldf/settings.yaml`:

```yaml
sail:
  implement:
    # Default crew members to dispatch
    default_crew:
      - coder
      - tester

    # Default reasoning level
    default_reasoning: medium

    # Default max tokens
    default_max_tokens: 16000

    # Auto-run tests after generation
    auto_test: true

    # Run guardrails automatically
    auto_guardrails: true
```
